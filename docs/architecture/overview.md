# PranaScan вҖ” Architecture

## Overview

PranaScan follows an **edge-first, privacy-by-design** architecture. Heavy signal processing
runs on the mobile device; the backend stores only anonymised metric snapshots.

## Data Flow

1. **Consent check** вҖ” App verifies active consent before any scan
2. **Quality gate** вҖ” On-device checks for lighting, motion, face confidence, audio SNR
3. **Scan session created** вҖ” Backend issues session ID
4. **On-device processing** вҖ” rPPG + voice analysis runs locally
5. **Result submission** вҖ” Only metric values (no raw video/audio) sent to backend
6. **Audit logging** вҖ” Every API call auto-logged (middleware), immutable
7. **Trend computation** вҖ” Backend compares to prior 7-day baseline, flags anomalies

## Privacy Guarantees

| Data | Handling |
|------|---------|
| Raw video frames | Never leaves device |
| Raw audio samples | Never leaves device |
| Metric values (HR, HRV, etc.) | Encrypted in transit (TLS), stored with user_id |
| user_id | Pseudonymous UUID вҖ” no PII required |
| Consent record | Append-only; deletion requests honoured after 30-day legal hold |
| Audit log | Immutable; no delete endpoints |

## Quality Thresholds

| Metric | Minimum | Reject reason |
|--------|---------|--------------|
| `lighting_score` | > 0.4 | Poor lighting |
| `motion_score` | > 0.95 | Excessive movement |
| `face_confidence` | > 0.8 | Face not detected |
| `audio_snr_db` | > 15.0 dB | Background noise too high |

## Scan Result Schema

```json
{
  "hr_bpm": 72.4,
  "hrv_ms": 45.2,
  "respiratory_rate": 16.1,
  "voice_jitter_pct": 0.42,
  "voice_shimmer_pct": 1.8,
  "quality_score": 0.91,
  "flags": [],
  "trend_alert": null
}
```

`trend_alert` values (when present): `"consider_lab_followup"` вҖ” never diagnostic language.
